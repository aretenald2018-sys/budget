# 오늘 번 돈과 와인포인트 홈/위젯 리브랜드 계획

## 요청 원문

현재 코덱스로 혼자 쓰는 가계부 앱을 개발해놓은 상태임. 목표는 소비억제를 하는 방향으로 가려고 만들었음. 그러나 심리학적인 관점에서 억제는 반동을 일으킬 가능성이 크므로 앱을 리브랜딩/리디자인하고 싶음.

휴대폰 바탕화면에 위젯형태로 구현할 예정인데, 내가 근래 6개월간 소비하던 일평균 소비 / 2주평균소비/ 월평균 소비액을 산출한 뒤에, 내가 거기에 미달하는 소비를 할 경우 그 차액만큼 내 스스로 포인트를 적립해서 마치 현금을 버는 것 같은 기분을 느끼게 해주고 싶음. 그리고 거기서 일부를 와인포인트로 적립해서, 한달에 12만 와인포인트가 쌓이게끔 설계하고 싶음. 그러면 난 그 쌓인 돈으로 한 달에 한 번 정도만 와인을 즐기는 거임.

이런 방향으로 기획해서 와이어프레임을 좀 짜봐줄 수 있어? 아 참고로 위젯 자체는 홈화면에 무언가를 구현하고, 그거를 따로 떼서 위젯화시킬 수 있는 방향으로 가면 좋겠고.. 홈화면에/ 아니면 이 앱에 어떻게 이 사상을 녹일 수 있을지 와이어프레이엄을 짜줘

## 이해한 내용

- 목표: 앱의 정서를 "소비 억제"에서 "내 평소 소비보다 덜 쓴 차액을 번 돈처럼 경험하는 보상형 자율 시스템"으로 바꾼다.
- 목표: 홈 첫 화면에 위젯으로 분리 가능한 `오늘 번 돈` 모듈을 만들고, 같은 모듈을 Android 홈 위젯 또는 native wrapper 위젯으로 재사용할 수 있게 설계한다.
- 목표: 최근 6개월 소비 기준선과 현재 소비를 비교해 일간/2주/월간 `획득 포인트`를 보여준다.
- 목표: 획득 포인트 일부를 `와인포인트`로 자동 배분하되, 월 120,000원은 보장액이 아니라 열심히 절약해야 닿는 상한/도전 페이스로 둔다.
- 비목표: 사용자를 꾸짖거나 특정 카테고리를 금지하는 문구를 만들지 않는다.
- 비목표: 실제 은행 잔액이 늘어난 것처럼 오해시키지 않는다. 회계상 실제 돈과 앱 내부 포인트는 분리한다.
- 사용자 흐름: 앱 실행 또는 홈 위젯 확인 -> 오늘/이번 2주/이번 달의 "번 돈" 확인 -> 와인포인트 진행률 확인 -> 필요하면 상세에서 산식과 거래 내역 확인 -> 월말에 실제로 모인 와인포인트 범위 안에서 한 번 즐기고 기록한다.
- 데이터 가정: 최근 6개월은 현재 날짜 기준 직전 180일 또는 직전 6개 완료 월을 기본으로 하되, 월간 기준선은 완료된 월만 쓴다.
- 열려 있는 질문: 와인포인트가 실제 계좌 이체/예산 잠금과 연결되어야 하는지, 아니면 앱 내부 보상 지갑으로만 남을지.

## 그릴 결과

- 적용 트리거: `/grill-me`
- 핵심 질문: `오늘 번 돈` 포인트를 실제 예산 계정처럼 엄격하게 다룰 것인가, 아니면 행동 강화용 보상 점수로 다룰 것인가?
- 추천 답변: 실제 지출/잔액과는 분리된 행동 보상 지갑으로 시작한다. 다만 산식과 원천 거래를 투명하게 보여줘 "내가 진짜 덜 쓴 차액"이라는 감각은 유지한다.
- 사용자 답변: 아직 없음.
- 확정된 결정: 추천 답변을 기본 설계로 삼는다.
- 남은 가정: 사용자가 원하면 후속 슬라이스에서 실제 계좌 이체 체크리스트나 월말 정산 루틴을 붙인다.

## 리브랜드 방향

### 기존 프레임

- "얼마나 썼나"
- "예산을 넘었나"
- "줄여야 하나"
- 실패 시 죄책감과 반동 위험이 큼.

### 새 프레임

- "오늘 평소보다 얼마를 벌었나"
- "이번 2주 동안 얼마를 지켰나"
- "와인 한 병을 즐길 권리를 어떻게 만들었나"
- 핵심 감정은 금지, 결핍, 채찍이 아니라 획득, 선택권, 의식적 즐거움.

### 추천 네이밍

- 앱/홈 정서: `오늘 번 돈`
- 포인트: `번돈포인트`
- 와인 지갑: `와인포인트`
- 월말 보상: `셀러 데이`
- 설명 문구: "덜 쓴 돈을 사라지게 두지 않고, 내가 볼 수 있는 보상으로 모읍니다."

## 포인트 산식

### 기준선

- 일평균 기준선: 최근 180일의 소비 합계 / 소비 데이터가 있는 일수.
- 2주 기준선: 최근 180일 일평균 기준선 * 14. 기존 앱의 격주 시작일 설정과 맞춘다.
- 월평균 기준선: 최근 6개 완료 월의 월 소비 평균. 현재 진행 중인 달은 기준선 계산에서 제외한다.
- 소비 합계에는 `card_payment`, `transfer_out` 중 `isBudgetExcluded(tx)`가 아닌 거래만 포함한다.
- 투자/저축/내 계좌 이동/환급 예정/정산용 거래는 기준선과 현재 소비 모두에서 제외한다.

### 획득 포인트

- 오늘 번돈포인트: `max(0, 일평균 기준선 - 오늘 소비)`
- 이번 2주 번돈포인트: `max(0, 2주 기준선 - 이번 2주 소비)`
- 이번 달 번돈포인트: `max(0, 월평균 기준선 - 이번 달 소비)`
- 진행 중인 기간에서는 "확정"과 "예상"을 구분한다.
- 오늘은 실시간 값으로 보여준다.
- 2주/월은 현재까지의 확정 차액과 기간 말 예상 차액을 함께 보여준다.

### 와인포인트

- 월 기준점: 120,000원. 이는 보장 적립액이 아니라 와인 한 번을 계획할 수 있는 도전 상한이다.
- 기본 원칙: 번돈포인트가 생길 때만 와인포인트가 쌓인다. 소비가 기준선을 넘은 날에도 와인포인트를 억지로 음수 처리하지 않는다.
- 배분 규칙: `winePoints = min(120,000원, 월 누적 번돈포인트 * wineAllocationRate)`로 계산한다.
- 배분율 산정: 최근 6개월 소비 추이에서 "상위 노력 월"에 해당하는 절약 가능액을 추정하고, 그 절약액을 달성했을 때 와인포인트가 120,000원 근처에 닿도록 `wineAllocationRate`를 잡는다.
- 예시 산식: `wineAllocationRate = clamp(20%, 70%, 120,000원 / highEffortMonthlyEarnedPoint)`.
- 기본 `wineAllocationRate`: 초기값 30%, 단 슬라이스 1에서 fixture로 보정 가능하게 둔다.
- 목표 보정: 월말 예상 와인포인트가 120,000원에 못 미치면, "보장 실패"가 아니라 "지금 페이스면 84,000원, 120,000원까지는 하루 평균 4,200원 더 벌면 됨"처럼 보여준다.
- 월 120,000원에 도달하면 추가 번돈포인트는 `자유포인트` 또는 `장기목표포인트`로 표시한다.
- 월말 와인포인트가 120,000원 미만이면 그 금액에 맞는 와인/잔술/다음 달 이월 선택지를 보여준다.

## 화면 톤

- 숫자는 초록/금색만 쓰지 않고, 기본 텍스트와 은은한 포인트 컬러를 섞는다.
- 초과 소비 상태도 빨간 경고 대신 "오늘은 적립 없음", "내일 다시 pace 회복 가능"처럼 복구 가능하게 표현한다.
- 와인 이미지는 과시적 보상보다 작은 셀러/라벨/잔의 느낌으로 쓴다.
- 홈 첫 컴포넌트는 카드처럼 보이되 앱 전체 안에서 과한 랜딩 페이지처럼 커지지 않게 한다.
- 위젯으로 떼어낼 수 있도록 고정 비율, 짧은 텍스트, 독립 상태 문구를 사용한다.

## 기존 코드 확인 결과

- `render-home.js`는 독립 홈 구현이 아니라 `renderReport({ rootSelector: '#tab-home', homeMode: true })`를 호출한다.
- `render-report.js`는 월/격주 거래를 함께 불러오고, 홈 모드에서 히어로/검토 카드/변동비 게이지를 렌더한다.
- 기존 격주 기준은 `utils/cycles.js`와 앱 설정의 `biweeklyStartDate`를 사용한다.
- 기존 소비 집계는 `aggregateByCategory()`, `isBudgetExcluded()`, `expenseTransactions()` 계열을 따른다.
- 와인 관련 데이터는 `wine_bottles`, `wine_tastings`, `openWineCellar`, `mindbank` 흐름이 이미 있다.

### 컴포넌트별 결정

- `reportModeControlHtml()`: 리포트 탭에서는 기존 `[이번 2주] [이번 달]`을 유지한다. 홈 모드에서만 `[오늘] [2주] [월]`로 확장한다.
- `.report-hero-head`: DOM 골격과 타이포 스케일은 유지한다. 금액 의미만 `currentUsed`에서 `earnedPoints.selected.amount`로 바꾼다.
- `.report-hero-progress`: 예산 소진률이 아니라 와인포인트 진행률을 보여준다. 다만 `tds-progress` 스타일은 그대로 재사용한다.
- `reviewNudgeCard()`: 위치는 유지한다. 검토 거래가 포인트 계산을 흔들 수 있으므로 "정확도" 맥락으로 연결한다.
- `.home-variable-panel`: 기존 변동비 게이지를 없애지 않는다. 사용자가 왜 포인트가 늘거나 줄었는지 내려가며 확인하는 원인 분석 영역으로 쓴다.
- `.fixed-cost-panel`: 보상 루프의 주인공이 아니므로 아래에 유지한다. 홈 첫 화면의 핵심 주의를 빼앗지 않게 한다.

## 예상 데이터 모델

- `settings.app.earningWidget`
  - `enabled: boolean`
  - `lookbackDays: 180`
  - `wineMonthlyTarget: 120000`
  - `wineAllocationRate: 0.3`
  - `baselineMode: 'rolling_180d' | 'completed_6_months'`
- 계산 결과는 Firestore에 저장하지 않고 클라이언트에서 거래 기반으로 산출한다.
- 월말 확정 포인트 기록이 필요해지면 후속으로 `users/{uid}/earning_snapshots/{monthKey}`를 추가한다.

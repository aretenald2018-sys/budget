# GPT 프로세스 개선안 취사선택 기록 (2026-07-24)

## 1. 제안 요약

사용자가 GPT로 정리한 "AI 기반 앱 개발 프로세스 개선안"을 이 프로젝트(바닐라 JS ESM SPA, 프레임워크·번들러·
TypeScript 없음, Firebase/Firestore 실데이터, Node 내장 테스트 러너)에 맞게 유효한 부분만 수용하고 부적절한
부분은 쳐냈다. 핵심 제약은 **사용자가 동작 사양을 객관식으로는 고를 수 있지만, 사전 지식이 없어 모든 동작을
미리 정의해둘 수는 없다**는 점이다. 따라서 "계약서를 사람이 먼저 다 쓴다"는 제안의 전제는 이 프로젝트에서
성립하지 않으며, AI가 초안+객관식 질문을 만들고 사용자는 고르는 방식으로 뒤집었다.

## 2. 수용 목록

- **화면 계약서(동작 SSOT)** — 단, **작성 주체를 역전**한다: 사람이 먼저 쓰는 게 아니라 AI가 코드·이미지에서
  초안을 채우고, 갈리는 동작만 객관식(추천안 포함)으로 묻고, 미답변은 AI 기본값을 `가정`으로 명시한다.
  (사용자의 명시 제약을 반영한 형태.) → `WORKFLOW.md` "화면 계약서" 단계, `contracts/TEMPLATE.md`,
  `contracts/home.contract.md`.
- **데이터 계약 + 계산/렌더 분리 + 테스트 선행** — 기존 `features/*/model.js` + `test/*.test.mjs` 패턴의
  형식화. → 계약서 §2 데이터 계약, `DEFINITION_OF_DONE.md` 코드·테스트 절.
- **상태 행렬**(로딩/데이터 있음/없음/오류/저장 중) — 정상 화면만 정의하면 나머지 상태가 빠지는 문제를 막는다.
  → 계약서 §3.
- **버그 안정화 우선** — 핵심 흐름이 깨진 상태에서는 신규 기능 계획을 잡지 않는다. 버그는 근본 원인 단위로
  묶는다. → `WORKFLOW.md` "버그 안정화 우선".
- **세로 슬라이스 구현** — 신규가 아니라 기존 "한 실행 세션 = 한 슬라이스" 규칙의 재확인. → `WORKFLOW.md`
  Planning/Execution 세션.
- **Playwright 브라우저 실조작 검수 + 시각 회귀 픽셀 diff CI 게이팅** — 사용자의 명시 지시로 포함. 320/360/
  390/412px. → `DEFINITION_OF_DONE.md` 검수·테스트 절, ROADMAP 묶음 D.
- **Definition of Done** — 프로젝트 실정판. 존재하지 않는 검증(타입 검사)은 기준에서 제외하고 lint/test/verify/
  Playwright만 기준으로. → `DEFINITION_OF_DONE.md`.
- **장식용 데이터 금지 / fixture 4조건** — `DEFAULT_MODEL`·폴백 스파크라인이 실제 위반 사례. → `WORKFLOW.md`
  "장식용 데이터 금지", ROADMAP 묶음 A.

## 3. 거부 / 축소 목록과 사유

- **새 `/docs` 트리(15+ 파일)** → 거부. 기존 `docs/ai` 구조에 통합한다(사용자 확정). 중복 문서 트리는 유지
  비용만 늘린다.
- **접근성/성능 테스트 스위트** → 보류. 1인용 앱이라 우선순위가 낮다. 필요 시 별도 백로그화.
- **AI 3세션 분리(Plan→Execute→Review)** → 신규 항목 아님. 이미 `WORKFLOW.md`에 존재하는 체계다. 제안을
  "신규 도입"으로 오기하지 않고, planning 세션에 계약 단계만 보강했다.
- **React 전제 용어(재마운트·URL 상태·컴포넌트 언마운트)** → 바닐라 JS 맥락으로 번역. "재마운트" 대신
  "영향 영역만 `innerHTML`/`outerHTML` 갱신", "URL 상태" 대신 "전환 시 스크롤 위치·입력값 보존"으로 옮겼다.
- **전면 TDD** → 축소. 계산 함수(model)만 테스트 선행하고, UI는 계약서 + E2E 스모크로 검증한다. 렌더 함수
  전체를 TDD로 감싸는 것은 이 코드베이스의 실익이 낮다.
- **사실과 다른 전제 제거** → "모달이 빈 껍데기", "그래프가 전부 가짜" 같은 전제는 현재 코드 기준 성립하지
  않는다. 기간 설정 모달은 실제 저장 동작이 있고, 라이브 경로의 그래프는 `buildTrend` 실데이터다. 이런
  전제에서 파생된 제안은 반영하지 않았다(진짜 문제인 폴백 잔재만 묶음 A로 남김).

## 4. 실사에서 확인된 실제 문제 (백로그로 연결)

`docs/ai/ROADMAP.md`의 근본원인 묶음 백로그로 옮겨 추적한다.

- **묶음 A(데이터 정확성)**: `features/home/dashboard.js`의 `DEFAULT_MODEL`(line 33–90)과 폴백 스파크라인
  배열이 운영 렌더 경로에 남아 있음. 데이터가 없을 때 샘플 금액·가짜 곡선이 실데이터처럼 노출됨.
- **묶음 B(상태/상호작용)**: 2주↔달 토글 미연결은 `0c3fdb1`로 수정 완료. 잔여는 세그먼트 전환 시 스크롤·입력
  보존을 E2E로 고정하는 것.
- **묶음 C(공통 구조)**: `.bottom-nav`가 4개 CSS 파일(`styles/60-shell.css`, `styles/00-foundation.css`,
  `styles/features/finance.css`, `styles/features/home-dashboard.css`)에 중복 정의됨. 홈 탭 전용 내비 변형
  (finance 숨김·색상)도 같은 슬라이스에서 정리.
- **묶음 D(반응형)**: 320/360/390/412px 시각 회귀 베이스라인 미구축. Playwright 슬라이스에서 생성 후 잘림·
  겹침 개별 이슈화.

이 묶음들은 이번 프로세스 검토 슬라이스에서 **수정하지 않고 백로그로만 남긴다.** 각 묶음은 착수 전 계획 문서를
만들고 한 슬라이스씩 진행한다.
